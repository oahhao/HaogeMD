# 开发执行纪律

## 变更边界

- 仅执行用户请求的变更，不主动扩展范围
- 修复 Bug 时优先定位根因，不通过注释代码、吞异常、绕过类型检查或添加无意义默认值掩盖问题
- 不主动引入新架构、新依赖、新工具链或新配置
- 如必须引入新模式，必须先说明原因、影响范围和迁移方式，等待确认后再执行
- 不得同时保留新旧两套重复逻辑，除非用户明确要求兼容过渡

## 代码复用

- 创建新实现前，必须先搜索项目中是否已有可复用代码
- 优先沿用现有目录结构、命名方式、调用链路和工具函数
- 有现成模式时，在现有模式上迭代，不轻易重写
- 发现重复逻辑时，先判断是否属于当前任务范围；超出范围只提出建议，不直接修改

## 环境意识

- 代码需区分 dev、test、prod 环境行为
- Mock、stub、fake data 只能用于测试或明确的本地开发场景
- 禁止在 dev 或 prod 逻辑中混入测试专用数据
- 禁止覆盖 `.env`、密钥、token、证书或 CI 配置；如确需修改，必须先说明原因并等待用户确认
- 不得把密钥、token、密码写入代码、日志、提交信息或文档示例

## 本地服务器管理

- 启动开发服务器前，应优先检查项目已有启动脚本和端口约定
- 启动新服务器前，必须确认是否已有相关进程正在运行
- 不得无条件启动多个重复服务器
- 如需终止旧服务器，必须先说明目标进程、端口和原因
- 测试完成后，应清理本次启动的临时进程
- 不得终止与当前任务无关的用户进程

## 测试与验证

- 修改完成后必须运行项目已有验证命令
- 优先使用项目内已有脚本，例如 test、lint、typecheck、format、build
- 不为追求覆盖率添加无意义测试
- 测试文件应放在项目既有测试结构中
- 无法运行验证时，必须说明原因，并标注「未验证」
- 不得声称“已通过测试”，除非确实执行过对应命令并获得结果

## 工具使用

- 需要最新信息、官方 API、版本差异或外部行为时，应优先查阅官方文档或项目内文档
- 使用外部工具前，必须确认其对文件系统、网络、进程或配置的影响
- 避免编写一次性脚本；确需编写时，应说明用途，并优先放在临时位置或遵循项目脚本目录约定
- 不在项目根目录执行可能生成非规范文件或目录的命令
