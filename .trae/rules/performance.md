# 性能优化规范

## 优化原则

- 先定位瓶颈，再进行优化
- 没有证据时不得假设性能瓶颈
- 优先优化算法复杂度、数据结构和 IO 次数，而不是做微小语法级优化
- 性能优化不得改变业务行为
- 性能优化不得牺牲错误处理、类型安全和可维护性

## 分析方法

- 优化前应明确性能目标，例如响应时间、内存占用、启动速度、吞吐量或 CPU 占用
- 能测量时必须基于实际数据、日志、Profile、Benchmark 或用户提供的复现场景判断
- 无法测量时，必须标注「待验证」或「需基准测试」
- 对复杂性能问题，应先给出分析路径，再修改代码

## 常见优化方向

- 避免重复计算，必要时使用缓存、记忆化或预计算
- 选择适合访问模式的数据结构
- 减少不必要的对象复制、深拷贝、序列化和反序列化
- 延迟计算直到真正需要时再执行
- 合并高频 IO、网络请求、数据库查询或 IPC 调用
- 大数据列表优先考虑分页、分批处理、流式处理、虚拟滚动或懒加载

## 并发与异步

- 识别可并行化任务前，必须确认任务之间没有共享状态冲突
- 并发优化必须考虑线程安全、锁竞争、取消机制和错误传播
- CPU 密集型任务不得阻塞 UI 线程或主事件循环
- IO 密集型任务优先使用语言或框架提供的异步原语
- 不得为了并发而引入不可控的任务泄漏、资源泄漏或顺序不确定问题

## 验证要求

- 修改后应运行项目已有测试或性能验证命令
- 如果没有现成验证方式，必须说明建议的验证方法
- 不得声称性能已提升，除非有可对比的数据或明确的复杂度改善依据
