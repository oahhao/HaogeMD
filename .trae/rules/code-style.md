# 代码可读性规范

## 命名约定

- 使用能表达业务含义的名称，避免只有技术含义的泛化命名
- 遵循当前项目和语言生态的命名规范
- 避免不必要的缩写、拼音、单字母变量
- 仅在局部循环、数学公式或语言约定场景中使用短变量名，例如 `i`、`j`、`x`
- 同一业务概念在项目中必须保持命名一致，不得随意切换同义词

## 代码组织

- 相关代码应放在一起，减少跨文件跳转成本
- 函数只承担一个清晰职责
- 同一函数内保持一致的抽象层次，避免高层业务流程和底层实现细节混杂
- 复杂条件判断应提取为具名变量或具名函数
- 公共逻辑抽取前，必须确认存在真实重复，禁止为假想复用提前抽象

## 注释与文档

- 注释解释“为什么”，不重复描述“做了什么”
- 对复杂业务规则、边界条件、兼容逻辑和风险点必须补充注释
- 公共 API、导出函数、核心类型应提供清晰文档
- 修改代码时必须同步更新过期注释和文档
- 删除或改写注释前，必须确认其不再表达有效业务约束

## 可读性边界

- 不得为了“看起来更优雅”改变业务行为
- 不得在未确认调用方影响的情况下重命名公共 API、导出类型或跨模块函数
- 不得为了减少行数牺牲可读性
- 不得把可读性优化扩展成无关重构
